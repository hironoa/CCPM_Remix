# 仕様・設計・実装計画・検証内容

## 回答：Excelやクリップボードからのデータインポート機能の実装案と実装内容
Excelなどからコピーしたデータ（TSV形式：タブ区切りテキスト）をペーストして取り込む機能を実装しました。

### 実装したUI
- 画面上部のヘッダー右端に「インポート」ボタンを追加しました。
- クリックすると、インポート設定とデータ貼り付け用のテキストエリアを持ったモーダルが開きます。

### データの形式（Excelからのコピー＆ペーストを想定）
モーダル内のテキストエリアには、以下の手順で無限階層に対応したデータを貼り付けることができます。
1. Excel（またはスプレッドシート等）で以下の列を持つ表を作成します。
   `No` | `タスク名` | `予定(h)` | `実績(h)` | `親No` | `先行No` | `担当者`
2. セル範囲を選択してコピーし、そのままモーダルのテキストエリアにペースト（`Ctrl+V`）します。（クリップボードの挙動として自動的に項目がタブ区切り（TSV）となり、そのまま当システムでパースできるようになっています）

## 回答：ガントチャート上での依存関係設定について
- **事象とご要望**: ガントチャートから直感的にタスク同士のつながり（先行・後続関係）をマウスで引けるようにする追加機能の実装と、ドラッグハンドルのUI・サイズについてご要望をいただきました。
- **対応内容**:
  1. **依存関係の描画UI**: 各タスクバー（親要素のフェーズは除く）に対し、右側に「接続元（後続タスクへ向かう矢印）」のハンドルを追加しました。
  2. **ドラッグ＆ドロップ機能**: ハンドルをドラッグして別タスクにドロップすることで、先行関係をグラフィカルに設定（`predecessors` にID追加）できるように実装しました。
  3. **視認性とサイズの調整**:
     - 単なる「青丸」から、中に「→（右矢印）」のアイコンを表示して役割を分かりやすくしました（サイズも `w-3` に大きく調整）。
     - ハンドルの高さはタスクバーの高さ（20px）に合わせて最適化し、スリムに整えました。
     - 「タスクの端（ハンドル）」と「接続ハンドル（青・緑丸）」の間には **2pxのクリアランス（隙間）** を設け、かぶりが全く発生しないように配置を再調整しました。
     - カーソルホバー時に青丸・緑丸が大きく膨らむアニメーション（拡大エフェクト）を廃止し、フラットな操作感に変更しました。
  4. **依存関係の削除機能**: 引かれた矢印線（Dependency Lines）をクリックすることで、依存関係を削除できるように対応しています。
  5. **閉路（循環）エラーの防止**: A → B → A のようなループ構造を作ってしまわないよう、ドラッグで結合した際に内部で再帰的にGraphを検証し、閉路を持つ接続は無視する保護回路を入れています。

### WBSビューからのネスト操作（インデント・アウトデント）
- **操作列の拡充**: WBSビューの右端「操作」カラムに、新しく「レベルを下げる（Indent）」と「レベルを上げる（Outdent）」ボタンを追加しました。
- これにより、行を上のタスクの子要素としてぶら下げたり、親要素から独立させて１つ上の階層に昇格させたりするネスト（階層）修正が、直感的なクリック操作で行えるようになりました。

### WBSビューのドラッグ＆ドロップによる階層・順序の変更
- **ドラッグ操作の対応**: WBS内でのタスクの表示順序の入れ替えや、親タスクの変更（子タスクにする）操作をドラッグ＆ドロップで実行できるように実装しました。
- タスク行の左端に表示される **グリップ（≡）アイコン** をドラッグすることで操作を開始します。
- ドロップ時の視覚的なフィードバックにより、**「要素の上（青い上線）＝前に挿入」「要素の下（青い下線）＝後に挿入」「要素の中央（青い枠線）＝子タスクとして挿入」** を直感的に選択・指定できます。

### PERT図でのグループ選択・まとめて移動
- **範囲選択と一括操作**: PERT図の背景をマウス（左クリック）でドラッグすることで、矩形の選択ボックスが表示され、複数のタスクをまとめて選択状態にできるようにしました。
- 選択された複数のタスクは、そのままドラッグすることでグループ単位で位置を一括移動（一斉配置）することが可能です。
- **キャンバスのパン（視点移動）操作**: 左クリックを「範囲選択」に割り当てたため、キャンバス自体の移動は「**右クリックを押しながらドラッグ**」または「**スペースキーを押しながら左クリックドラッグ**」の標準的な操作に移行しています。
- タスク選択時は青色の境界線（リング）でハイライトされる視覚的なフィードバックを強化しました。
- **ノードの余白縮小**: ブロック（各タスクノード）内部のパディング（余白）を `p-4` から `p-2.5` 〜 `p-3` 相当に絞り、コンパクトな見た目に最適化しました。

### 無限のネスト階層への対応
- 「グループ名」の代わりに **`親No`** 列を導入しました。
- これにより、「タスクの `親No` に別のタスクの `No` を指定する」という自己参照構造の組み合わせで、いくらでも深い階層（親＞子＞孫＞ひ孫...）を表現することが可能です。

### 機能の特徴
- **自動ID・依存変換**: `1.1` や `A-1` などの人間が読みやすい「No」を内部で安全なUUIDに自動変換しつつ、その「No」を使った依存リンク（親子や先行タスク）の関係を正しく復元します。
- **データのリセット / 追記**: 「現在のデータを全て上書きリセットする」機能と、「現在のデータは維持したまま、新しいタスクとして末尾に追加（追記）する」機能を選択可能にしています。追記の場合は、新規タスクの「先行No」に既存のタスク名を指定して連結させることもできます。

## 回答：バッファが13hとなる計算式について
- **ご認識の通り、バッファは対象となるタスクの見積工数から算出されるのが正しい挙動です。**
- 現在のシステムでは、CCPM（クリティカルチェーン・プロジェクトマネジメント）の原則に則り、プロジェクトバッファおよび自動挿入される合流バッファの期間を以下の方程式で算出しています。
  - **合流バッファの計算式**：  
    `合流バッファ工数 = 先行する非クリティカルチェーンの見積工数（duration）の合計 × 設定されたバッファ比率（デフォルト50%）`
  - つまり、対象となる箇所の合流バッファが「13h」と表示されている場合、その直前に繋がっているタスクチェーンの見積工数の合計が **26時間** であり、その50%として **13時間** のバッファが算出・確保されたことを意味します（26h × 0.5 = 13h）。
  - （※親グループの合算値ではなく、依存関係で繋がった「チェーンの長さ」そのものに対して計算されます）。

## 回答：数値のフォーマット統一
- **要求**: ガントチャートのリストの小数点第2位で丸めてください。ただし、0は省略してください（例: 10.50 -> 10.5, 10.00 -> 10）。
- **対応**: 
  - `GanttView.tsx` 内の表示においてヘルパー関数 `formatNum = (num) => Number(num.toFixed(2))` を定義しました。
  - リスト左側の「進捗(%)」「予定(h)」「実績(h)」「残(h)」のすべての表示と `<input>` の `value` に対してこのフォーマットを適用しました。JavaScript の `Number()` 変換の特徴を活かすことで、小数点第2位まで保持しつつ不要な末尾の `0` を自動的に省略することができています。
  - 同時に、ガントバー内の進捗率表示やバッファ消費量のオーバーレイ表示についても同様に小数点第2位までの丸め処理を一貫して適用しました。

## 回答：ガントチャートがずれる現象の原因と対応

## 回答：保存時のネットワークエラーの修正（Failed to fetch）
- **原因と事象**: 以前導入した `saveData` のデバウンス処理によりネットワークへの連続アクセスは防げていましたが、PERT図コンポーネント（React Flow）にて「ドラッグ中」の `onNodesChange` イベントが発火するたびに `updateTask` を毎フレーム呼び出してしまっていました。これによりZustandストアが毎秒60回更新され、Reactの再描画（リレンダリング）ループやReactFlowの内部ノードの強制上書きが頻発しました。結果としてメインスレッドがフリーズ（ハングアップ）状態になり、完了待ちだった `fetch` リクエストがブラウザによって異常切断され `Failed to fetch` エラーが発生していました。
- **対処**: PERT図の `onNodesChangeImpl` において、`dragging: true`（ドラッグ中）の最中はストアの `updateTask` を呼び出さず、`!c.dragging`（ドラッグが完了した瞬間）にのみタスクの `position` を一度だけストアに保存するように修正しました。これによりメインスレッドの過負荷が解消され、ネットワークエラーも解消しました。

## 回答：自動計画作成機能（自動スケジューリング）の実装
- **要求**: PERT図とWBSの依存関係や見積工数、さらにリソースのアサイン状況に基づいて、自動で計画（開始日などのスケジューリング）を作成したい。
- **実装機能**: 画面のヘッダー（「インポート」ボタンの横）に **「自動計画」ボタン（魔法の杖アイコン）** を追加しました。
- **自動計画のアルゴリズム（処理内容）**:
  1. 現在設定されている手動の開始日時（`manualStartDate`）を一旦すべてリセットします。
  2. WBSやPERT図で定義された先行・後続の依存関係（Graph）を解析し、トポロジカルソートを実行します。
  3. 各タスクを以下の優先順位で選出・割当を行います。
     - スラック（許容遅延時間＝クリティカル度）が小さい（よりクリティカルな）タスクを優先。
     - 見積り工数が大きいものを優先。
  4. 選ばれたタスクに対して、**すべての先行タスクが完了する日時** と、**割り当てられた担当者（リソース）の他のタスクが終わって空きができる日時** の両方をシミュレーション計算し、「**依存関係が満たされており、かつ担当者が空いている最短の日時**」を開始日として自動で割り当てます。
  5. 最速でプロジェクトが完了するよう、担当者の重複（リソース競合・被り）を回避しながらスケジュールが平準化（山崩し）されて再描画されます。

## 回答: WBS表示項目メニュー外部クリック修正
- **要求**: WBSの表示項目のメニューも外部クリックで閉じるようにして下さい。
- **原因と対処**:

- **要求**: ガントチャートなどで進捗や実績工数、残工数を更新した際、すべての項目が連動して自動計算されてしまうが以下の通りの挙動にして欲しい。
